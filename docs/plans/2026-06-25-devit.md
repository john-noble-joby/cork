# devit Skill Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. NOTE: devit is a single Markdown skill file — inline execution is likely more appropriate than per-task subagents (see Execution Handoff).

**Goal:** Add a `devit <STORY>` Claude Code skill to the cork repo that runs the full Linear-story dev loop by orchestrating the existing `cork` and `copilot-review-loop` skills plus the superpowers workflow skills.

**Architecture:** devit is a session-driven skill — `skills/devit/SKILL.md` is a set of instructions the active Claude Code session follows. It adds the front-end (Linear verify, story-size gate, worktree/branch setup) and conventions (branch naming, PR format, pushback surfacing), and delegates implementation to `writing-plans`+`subagent-driven-development`, review to `cork`, and PR review to `copilot-review-loop`. No `orchestrate.py` change.

**Tech Stack:** Markdown skill file; `git worktree`; Linear MCP; `gh`; existing cork skills. Deployed via `install.sh`.

## Global Constraints

- **No `orchestrate.py` / Python changes** — devit is a skill (Markdown) that drives existing tooling.
- **Branch naming:** `feature/<TICKET>-<slug>` for features, `bugfix/<TICKET>-<slug>` for bugs (`<slug>` = short kebab-case of the story title).
- **PR:** title and body start with `<TICKET>: `; body always includes an **"In plain terms"** section (what it does / adds / removes) + the Linear ticket link.
- **Story-size target:** ≤500 diff lines/branch; cork degrades ≥1,500, fails ≥5,000. Split on: multiple runtimes/layers · new type + all consumers · >3 new test files.
- **Story-split flow:** propose split → **user verifies/adjusts** → devit writes sub-stories to Linear → do smallest slice. (Human-verify gate before any Linear write.)
- **No auto-merge:** devit ends at "PR through the loop + pushbacks surfaced." The human merges.
- **Human-in-the-loop:** any agent may pause for clarification at any phase.
- **Base branch:** `develop` (overridable).
- **Skill versioning:** `skills/devit/SKILL.md` carries a `**Version:**` stamp synced to repo `VERSION`; `install.sh` checks all skill stamps == `VERSION`.
- Spec: `docs/specs/2026-06-25-devit-design.md` (source of truth for section content).
- **HELD:** do not execute until the multi-provider rework (PR #1) merges to `main`.

### Task 2: Author Phases 0–3 (verify → size-gate → setup → implement)

**Files:**
- Modify: `skills/devit/SKILL.md` (fill Phases 0–3)

**Interfaces:**
- Consumes: the skeleton from Task 1.
- Produces: the front-half of the loop. Phase 4+ (Task 3) assumes Phase 2 has created a worktree at `.worktrees/<branch>` on a branch named `feature|bugfix/<TICKET>-<slug>`, and that `{TICKET}`/`{BASE}`/`{WORKTREE}` are known.

- [ ] **Step 1: Write Phase 0 — Verify the story**

```markdown
## Phase 0 — Verify the story

Fetch the story with the Linear MCP tools (by `<TICKET>`). Read title, description,
acceptance criteria, type/labels, and links.

- **Clarity gate:** if scope or acceptance criteria are unclear, ambiguous, or
  missing, **STOP and ask the user** before doing anything else. Do not guess.
- **Type:** classify feature vs. bug — Linear issue type/label first; else infer
  from content ("bug", "fix", "regression", an error report). This decides the
  branch prefix in Phase 2.
```

- [ ] **Step 2: Write Phase 1 — Story-size gate** (verbatim constraints + split flow)

```markdown
## Phase 1 — Story-size gate (before any code)

**Target ≤500 diff lines per branch.** cork's review degrades above ~1,500 lines
and fails hard above ~5,000 (model context overflow). Estimate from the story's
scope — you can't measure lines yet, so judge.

**Propose a split BEFORE implementing if the story requires:**
- changes across **more than one language runtime or architectural layer** (e.g.
  backend + frontend, service + its client, API + schema + UI);
- **a new domain type AND all its downstream consumers** (parser, schema, resolver,
  handler/dispatch, fixtures) — naturally two stories: (a) the type + its
  definition/parsing/schema, (b) the consumers + fixtures;
- **more than ~3 new test files.**

**If too big — split flow (human-verify gate):**
1. **Propose** a split: a list of sub-stories, each a title + one-paragraph scope,
   and which is the smallest complete, mergeable slice to do first.
2. **Wait for the user to verify/adjust.** Do not write to Linear yet.
3. **After confirmation, write it to Linear** via MCP: create the new sub-stories
   (and/or adjust existing ones), linked to the parent.
4. Proceed with the first slice as the active story for the rest of the run.
```

- [ ] **Step 3: Write Phase 2 — Setup** (worktree + branch, exact commands)

```markdown
## Phase 2 — Setup (worktree + branch)

Base is `develop` (override if the user says otherwise). Derive `<slug>` as short
kebab-case from the story title. Prefix `feature/` (or `bugfix/` if Phase 0 found a
bug). All work happens in the worktree, not the main checkout.

\`\`\`bash
git fetch origin develop
BR="feature/<TICKET>-<slug>"   # or bugfix/<TICKET>-<slug>
git worktree add ".worktrees/$BR" -b "$BR" origin/develop
cd ".worktrees/$BR"
\`\`\`

**Session naming (manual — Claude Code has no programmatic rename):** after creating
the branch, print this for the user to paste so the session is identifiable in
`/resume`:
`Tip: run  /rename <TICKET>-<slug>  to label this session.`
(devit cannot run /rename itself — it's a manual slash command with no tool/hook/API.)

Confirm with the user before starting work:
`devit: <TICKET> | <BR> | worktree .worktrees/<BR> | base develop — start? (split needed: yes/no)`
```

- [ ] **Step 4: Write Phase 3 — Implement** (decomposable→SDD, atomic→inline, during-size check)

```markdown
## Phase 3 — Implement

- **Decomposable story** (independent tasks): use `writing-plans` to draft a short
  plan, then `subagent-driven-development` to execute it — fresh subagent per task,
  **dispatched in parallel where tasks are independent**, with a review gate between.
- **Atomic/small story:** implement inline in the session.
- **During-implementation size check:** if the diff will cross ~500 lines, STOP and
  flag the user. Propose the smallest complete, mergeable slice; file the remainder
  as a follow-on Linear story (same propose → verify → write-to-Linear gate as
  Phase 1). Don't silently blow past the target.
- Run the repo's tests before moving on.
```

- [ ] **Step 1: Write Phase 4 — cork review + fix**

```markdown
## Phase 4 — cork review + fix

Run the usual cork **full** review→fix flow on the branch (invoke/follow the `cork`
skill): per-model blind review → apply the valid findings or **push back with
justification** → commit after each model. Record every pushback for the Phase 7

```

- [ ] **Step 2: Write Phase 5 — Open the PR** (title + "In plain terms")

```markdown
## Phase 5 — Open the PR

Push the branch and open a PR with `gh`:
- **Title** starts with `<TICKET>: ` — e.g. `MXE-123: Add per-station backdoor routing`.
- **Body** MUST include an **"In plain terms"** section: what this PR **does / adds /
  removes**, in non-jargon language. Follow with a short bullet list of what each
  review pass caught, and the Linear ticket URL at the bottom.
- Base branch: `develop`. Not a draft.
```

- [ ] **Step 3: Write Phase 6 — Copilot review loop**

```markdown
## Phase 6 — Copilot review loop

Run the `copilot-review-loop` skill on the PR. For each addressed item: leave a
reply comment and **mark the thread resolved**. Where a finding is wrong or
out-of-scope, **push back with justification** and resolve. Record pushbacks for
Phase 7. (The loop already handles request → poll → fix/push-back → re-request up
to its max passes.)
```

- [ ] **Step 4: Write Phase 7 — Finish + Notes**

```markdown
## Phase 7 — Finish (surface pushbacks)

Print a final summary:
- PR URL + branch.
- What each review pass (cork models + Copilot) caught.
- **Every pushback** (cork + Copilot) with its justification, grouped together so the
  human can scan them.

**Do NOT merge.** devit ends here — the PR is through the loop; the human decides on
merge.

## Notes

- **Human-in-the-loop:** at ANY phase, if something is unclear or risky, pause and
  ask the user. Clarification beats guessing.
- **Worktree cleanup** is the user's call (the PR branch worktree stays until they
  merge/close). Don't remove it automatically.
- **Path config:** skills/orchestrator come from `$CORK_HOME` (default `~/dev/cork`).
```
